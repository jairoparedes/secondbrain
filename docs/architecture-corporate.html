Resumen ejecutivo

El usuario accede desde el navegador a http://localhost. Todo el tráfico entra por Nginx, que actúa como único punto de entrada. Las páginas y recursos estáticos se sirven desde Next.js; las peticiones a /api/* se enrutan a Laravel, que autentica al usuario con un token Bearer de Sanctum.

Laravel se apoya en tres servicios de datos: PostgreSQL como fuente de verdad (incluyendo búsqueda vectorial con pgvector), Redis como caché, almacén de sesiones y cola de trabajos, y MinIO como almacenamiento compatible con S3 para adjuntos y exportaciones.

Los trabajos largos (generación de embeddings, reindexado, envío de mails) se delegan al Queue Worker, que los consume de Redis de forma asíncrona. La integración con OpenAI (Fase 4) es opcional y consciente de la ruptura del modelo zero-knowledge.

Flujo de un request: crear una nota
#ActorAcción
1NavegadorEl usuario escribe una nota y la aplicación envía POST /api/notes con el token Bearer en el encabezado Authorization.
2NginxReconoce que la ruta empieza con /api y reenvía la petición al backend por FastCGI al puerto 9000.
3LaravelEl middleware auth:sanctum valida el token contra personal_access_tokens. Si falla, responde 401.
4LaravelValida el payload (content_ciphertext, tag_ids, etc.) y verifica que los tags pertenecen al usuario.
5PostgreSQLLaravel ejecuta un INSERT en notes y sincroniza la tabla pivote note_tags.
6RedisOpcionalmente, Laravel encola un trabajo (p. ej. generar embedding) en la cola default.
7LaravelDevuelve 201 Created con el recurso completo: id, campos cifrados y lista de tag_ids.
8Queue WorkerEn segundo plano, consume el trabajo, llama a OpenAI (Fase 4), guarda el vector resultante en embeddings con pgvector.
